# EXPERIENCES — token prior for layout generation (Training-Free GRPO)

This is the **experiential knowledge library** in the sense of Training-Free GRPO
(arXiv 2510.08191): a frozen LLM (**Claude is the generator/policy**) reads these
natural-language experiences and CONDITIONS its generation on them — no fine-tuning.

**How to use (Claude = the generator, NOT code):** when asked to generate a layout,
DO NOT default to a solid parametric silhouette. Generate the layout (cell-set or build
recipe) by FOLLOWING every experience below. Priority = aesthetics + symmetry (the 4-axis
`geom.sym_scores`) + v3-solvable. Update this file by hand from what each composed layout teaches
(the auto GRPO loop was removed with the bulk pipeline).

Each entry: `[id] (option, refs) rule`. Learned by contrasting winner vs loser rollouts.

## Experiences

**[1]** (modify, round-7+9) **Build SPARSE-DEEP, never solid-shallow or over-fragmented.** A real board = a SMALL thin footprint (~16 base cells), MOSTLY single scattered cells (70% of base clusters are 1 cell), towered ~6 layers with **+0.5 diagonal stagger**. Total ≈72 cells, **n_clusters ~5–6 (CAP 12 — never above 12)**. Two failure modes: (a) solid silhouette (fill ~0.9, 1 cluster — R7 loser base_fill=0.944) and (b) over-fragmented (n_clusters=20 with cells=138 — R9 loser). Real boards are scattered BUT not atomized — 5–6 clusters of 2–4 cells each is the target.

**[2]** (add) **Recognizable / elongated shapes (sword, key, arrow, fish, leaf…) go DIAGONAL + sparse + deep.** Render the shape's PATH as a thin DIAGONAL line of sparse cells with a few crossbars, towered deep — NOT a solid vertical bar. Proof: the competitor's sword = 63 cells, **8 layers, diagonal**, scattered cells. A vertical solid sword (132 cells, 2 layers, aspect 0.47) is a LOSER; the diagonal sparse-deep sword (54 cells, aspect 0.85) is a WINNER. The shape reads from the sparse path + crossbars, not from fill.

**[3]** (modify, rounds 1+2+5) **Match these proportions — HARD FLOORS AND CAPS.** Held-out medians: width ~6 (4–9), height ~7 (4–10), **aspect w/h ~0.88**, **layers ~6 (FLOOR 4, CAP 8 — NEVER below 4 or above 8)**, **cells ~72 (range 40–100, NEVER above 100)**, capacity ~24, **base fill ~0.46**, **h-symmetry ~64%**. Evidence: R1 loser n_layers=10 → layout_diff=16 (cap violation); R2+R6 loser n_layers=3 → layout_diff too low (floor violation); R5 loser cells=141 → layout_diff=15 (cell cap violation, real p90=102). Build symmetric ≈2/3 of the time.

**[4]** (add) **Difficulty profile: deep-but-easy.** layout-difficulty ≈ 0.9 × layers (so ~6 layers → diff ~5–6). Cover ratio ~0.84 (most cells buried). ~12 cells pickable at start. This is ONLY achievable with sparse bases — solid bases at the same depth overshoot difficulty ~4×.

**[5]** (add) **Validity (hard).** Every cell at L>0 must OVERLAP ≥1 cell directly below (game cover rule, |dx|<1 & |dy|<1; corner contact at 0.25 area is OK — single-cell +0.5 towers are valid, don't require 0.5-area support). Total cells %3 == 0. Must pass v3-solvable.

**[6]** (modify, rounds 1+4) **Towers THIN but not TOO TALL — target tower_mean ~4 (range 3–5).** Use anchor heights **4–7 (a few at 3, most at 5–6)**; vary for thinning. HARD CONSTRAINT: mean tower height across all anchors MUST land 3–5 (target 4). R4 loser: tower_mean=11.79 (nearly every anchor reached near-max depth) → layout_diff=24 (5× overshoot). Vary heights so dense columns never form — if base has N anchors, expect tower_mean ≈ 0.65×max_height after variation.

**[7]** (add, from round-1) **Aspect: orient slightly TALL.** Round-1 symmetric scatter gave aspect 1.0 (real 0.88). Make the footprint ~1 row taller than wide (e.g. 6 wide × 7 tall) so aspect ≈0.88.

**[8]** (add, duck aesthetics session) **For RECOGNIZABLE SHAPES (icon/animal/logo): DECOUPLE silhouette from depth, and build depth with `capped_inset` — NEVER `compose()` isolated +0.5 towers.** Proven by diagnostic: a hand-traced duck's FLAT base-layer (L0) silhouette read perfectly as a duck, but the *stacked* render looked muddy — the fault was the depth model, not the mask. `compose()` gives every base cell a lone +0.5 tower, so the rim fringes into a checkerboard and the shape dissolves. `capped_inset` (in `layout_builder`, via `gen_shape_layout.py`) only stacks where a cell is supported by the interior, so the RIM stays 1 layer (silhouette crisp) and depth insets toward the centre (reads as a clean mound). Ranking observed (same duck mask, 5 layers): **capped_inset (clean) ≫ compose isolated towers (fringed) > uniform_stagger (309 cells, whole shape buried, no contrast)**. Workflow: (a) trace/import → 2D mask; (b) **verify the FLAT silhouette renders recognizably** (`gen_shape_layout.py` emits `_<name>_flat.png`) BEFORE adding depth — every earlier shape was wrongly judged on the depth render that fights the shape; (c) `gen_shape_layout.py --mask … --layers 5 --target N` builds capped_inset + peels exposed top cells (never floats, never touches base) to an exact cell count. Tension to SURFACE, not swallow: accurate shape → more base cells → at a fixed total/5-layers the towers go shallow (duck 67 base → avg height ~2.7); nail the silhouette first, then show the user the depth that a given total/layers leaves and let them pick.

**[9]** (modify, xmas-tree session) **DEFAULT to EVEN layer distribution; it's OK to ask for / use MORE tiles to keep depth even (USER PREFERENCE).** Depth DISTRIBUTION is a dial separate from depth amount: `capped_inset` concentrates hidden layers in a central MOUND (wide interior stacks tall; thin features — a tree's star/tiers/trunk — stay flat 1-layer), whereas EVEN spreads hidden layers across the WHOLE silhouette. **`gen_shape_layout.py` now defaults to EVEN** (uniform_stagger build + peel the TALLEST exposed cell first → levels tower heights); pass `--mound` only when a sculpted centre-mound is explicitly wanted. Proof: a 180/5 tree via capped_inset left the star+ngọn flat and dumped depth low-centre → user said "ưu tiên dàn đều layer, có thể yêu cầu thêm tile"; even mode gave ~2-3 layers across the entire tree. **The tension from [8] (more base cells → shallower at a fixed total) is now resolved IN FAVOUR of even depth: when the requested total is too low to give even depth at the target layer count, RAISE the tile total (suggest a number to the user) rather than concentrating depth or dropping layers.** Budget rule of thumb for even depth: total ≈ base_cells × 3–3.6 (e.g. 64-base tree → ~210–230 tiles for even 5-layer depth; at only 180 it stays even but thinner). Both peel modes remove only EXPOSED cells (never floats) and never touch layer 0 (silhouette stays crisp).

## Losers (anti-patterns — what to NOT generate)
- n_layers > 8 → tower_mean spirals → layout_diff overshoots 3×. [contradicts [3]]
- n_layers < 4 → layout_diff too low, board too shallow (R2, R6, R8). [contradicts [3]]
- tower_mean > 6 → layout_diff overshoots 5× even at moderate depth (R4/R10/R11). [contradicts [6]]
- cells > 100 → layout_diff overshoots (R5: 141→diff=15; R9: 138 + n_clusters=20). [contradicts [3]]
- n_clusters > 12 → over-fragmented base (R9 loser: 20 clusters). [contradicts [1]]
- base_fill > 0.7 → too dense / solid silhouette (R7 loser: fill=0.944). [contradicts [1]]
- Shallow 2-layer board → real is ~6 layers. [contradicts [1],[4]]
- Vertical thin bar for elongated shape → aspect ~0.5 (real 0.88). [contradicts [2]]
- Every layer = full base footprint → cells/difficulty overshoot. [contradicts [6]]

**[10]** (add, region-depth session) **Per-COLOUR-REGION depth: different colour regions can get different layer counts (e.g. a duck's beak/eye/mouth stacked DEEPER than the body).** `gen_region_depth.py --img <colour png>` reads the pixel art with PIL (sample each grid cell centre, classify colour → region), assigns a per-region tower height (`HEIGHTS={"Y":2,"O":4,"B":5}` = body thin / beak mid / eye+mouth deepest), and builds per-cell towers via `compose` (isolated +0.5 towers ARE valid — a 1-cell eye towered to 5 stands by the corner-overlap rule). Grid geometry is **MANUAL** via `--x0/--y0/--cell/--n` (there is **NO auto-detect** — read the image, eyeball offsets/cell-size; worked: x0=24,y0=24,cell=89,n=13 for the Prohama duck); per-region heights via `--heights "Y:2,O:4,B:5"`. PIL is imported lazily (clear error if missing — it's the only non-stdlib user). Symmetry is now **MEASURED + recorded** (`symmetry_axes/score`); pass `--mirror` to snap a front-facing/symmetric subject to 1.00 (default measures only, does not force). Renders `_<name>_regions.png` (flat colour map — verify segmentation) + `_<name>_stack.png` (layer depth). This INTENTIONALLY overrides the even-layers default [9]: depth varies by region here. Use when the request ties depth to colour/feature ("mỏ/mắt dày hơn").

**[11]** (add, soda-cup cover% session) **Hitting an exact "% completely covered" (cover100) target — mechanics + how.** "Covered 100%" = engine `DifficultyScorer.cover100_by_area(board, active, threshold=0.9)`: a cell with ≥90% of its 1×1 area hidden by higher-layer cells. With the +0.5 alternating stagger the dominant way a cell hits ≥90% is a **same-(x,y) cell exactly 2 layers above** (full overlap), or **4 corner cells on the layer above** (4×0.25=1.0). Consequences for a per-anchor tower of height h: cells at L≤h-3 are self-covered by their +2 partner; the **top 2 cells of every isolated tower are exposed** → an isolated h=5 tower is only 60% covered, but an **interior h=4 tower is ~100% covered** (its top L3 is capped by neighbours' L4). So: more height-4 interior towers + sparse height-5 caps ⇒ higher cover%; isolated/thin features ⇒ lower. To hit an EXACT ratio at fixed total+layers, run a **two-phase tuner** (see `gen_soda_cup_cov.py`): phase-1 converge total→target cells (5 layers); phase-2 **total-neutral swaps** (h[a]+1 on an interior tower, h[b]−1 on an edge/short tower), **re-measuring cover100 with the real scorer each step**, hill-climb to the exact count. Caveat learned: 50% is easy (near the natural value), but **75% needs a dense block of mostly height-4 towers + few height-5 caps** — a footprint that's too sparse or has thin protrusions (straws, masts) caps out (~67% observed) and can't reach 75%.

**[12]** (modify, symmetry-method) **Symmetry MUST be GEOMETRIC cell-mirror, NOT tower/base-anchor mirror.** For the VERTICAL axis that is `(L,x,y)->(L,-x,y)`; gen-layout now ranks 4 axes (`geom.sym_scores`): vertical `(L,-x,y)`, horizontal `(L,x,-y)`, diag `(L,y,x)`, anti-diag `(L,-y,-x)`. CORRECTION (user: "vẫn chưa đối xứng"): an earlier version mirrored at the TOWER level (mirror base anchor, re-render the +0.5 stagger) — that leaves odd-layer cells leaning the SAME way on both halves, rendering as a visible "thiếu 1-2 ô". MEASURED ground truth: on 213 real exact-tower-symmetric boards, GEOMETRIC cell-mirror symmetry is median **1.00** (odd cells DO flip lean: a +0.5 cell on the left mirrors to -0.5 on the right). So when you want a symmetric shape, BUILD it by rendering ONLY the dominant half + axis cells then adding their geometric mirrors (`geom.geom_symmetrize(cells, axis)`), and trim to %3 by dropping GEOMETRIC mirror PAIRS of exposed tops (or one axis top for the %3==1 case — never a single off-axis cell) via `geom.geom_div3_trim`. Verify with `geom.is_geom_sym` / `sym_scores` (intended axis should read **1.00**). This is `compose(spec, mirror=True, axis=...)`. Bulk symmetric generation was retired — compose ONE symmetric layout at a time (quality > a pile).

**[13]** (add, image-audit session) **SIMPLIFY a complex image BEFORE building — the representation space is SMALL — and the image path now MEASURES symmetry like compose.** (a) **Simplify-first / fidelity budget:** real boards carry only ~16–24 base footprints, ~6 layers, portrait aspect ~0.88. A detailed picture **cannot** be reproduced faithfully — reduce it to a low-res recognizable silhouette, drop small features; "reads like the idea" > pixel-faithful. `gen_shape_layout.py` now **auto-runs the complexity gate** (`evaluate_icon`) and WARNs over budget (>~48 footprints / aspect >1.1) — heed it by simplifying, NOT by adding tiles (this BOUNDS the earlier [8]/[9] "use more tiles" advice: raise tiles only for even depth, never to chase detail past the budget). (b) **Image-path symmetry (B-audit fix):** the shape path used to break symmetry of a symmetric mask — the +0.5 stagger erodes even layers (V≈0.92) and the old `trim_to_mult3`/`peel` dropped single off-axis cells. Now `gen_shape_layout.py`/`gen_region_depth.py` MEASURE `sym_scores` + record metadata, and `--mirror [--axis]` snaps to exactly 1.00 via `geom.geom_symmetrize`+`geom.geom_div3_trim` (center FIRST so the axis is at x=0), `--no-mirror` measures without forcing (same "đo & xếp hạng, không ép" as compose). `--min-sym` gates. (c) The SVG→mask parser was hardened: single-quote attrs, `transform`, and `fill:none`/stroke are now handled — before, these silently produced an empty or garbled mask. (d) **MATCH THE SOURCE OBJECT'S SYMMETRY: for every image, COUNT its reflection axes and build the layout with the same** — 0→`--no-mirror`, 1→`--axis vertical|horizontal`, 2→`--axis vh`, 4→`--axis d4`. CORRECTION to my earlier assumption: I thought the directional +0.5 stagger FORBADE exact diagonal/4-axis symmetry (odd layers lean one way). MEASURED ground truth: EXACT 2-axis (vh) and 4-axis (D4) ARE achievable and VALID — union the reflection-group orbit (D4 = 8 images, vh = 4), then REPAIR orbit-wise: the stagger floats some image cells, so drop each floating cell WITH its whole orbit (preserves the symmetry) until support is clean, then trim %3 by exposed-top orbits. Proven on a flower mask: D4 union alone → all 4 axes 1.00 but `structural_ok=False`/`div3=False` (floating); after orbit-repair → all 4 axes **1.00, structural_ok=True, div3=True**, n=138. (`geom.d4_symmetrize` / `geom.vh_symmetrize`; `gen_shape_layout.py --axis d4|vh`.) TRADEOFF: the D4 layout is shallower (cover ~0.45 vs ~0.56 for 1-axis) so its score ceiling is lower and the score band is bimodal — to hit a mid "Normal" score, sweep color_count around the jump (cc 13 hit 34; cc≤12→~9, cc≥16→~76).
